# Postgres Account System Spec

## Status

Proposed.

This spec supersedes `004-account-login-system.md` by grounding the account system in the
repository's current Next.js, Postgres, and session-ownership implementation.

## Repo Findings

- The app is a Next.js App Router workspace in `apps/web`.
- Protected API routes currently call `authenticateRequest` from `apps/web/src/server/auth.ts`.
- Current production auth supports either a shared `API_AUTH_TOKEN` or JWT validation through
  `AUTH_ISSUER`, `AUTH_AUDIENCE`, and `AUTH_JWKS_URI`.
- Current local auth can be disabled with `AUTH_DISABLED=true`, using request headers or
  `local-dev-user`.
- The browser currently stores validated auth credentials in `sessionStorage`.
- `db/migrations/001_postgres_pgvector.sql` creates `onboarding_sessions` with an `owner_id` text
  column, and `PostgresSessionRepository` already scopes session reads and writes by owner ID.
- `db/migrations/002_users_table.sql` creates a basic `users` table, but the table is not used by
  runtime authentication yet.
- `scripts/users-create.mjs` hashes passwords with bcrypt, but it writes to `data/users.json` rather
  than Postgres.

## Goal

Add an internal account system backed by Postgres so users can sign in with email and password,
receive a secure server-side browser session, and access only their own onboarding data.

## Non-Goals

- Public self-service registration.
- OAuth, SAML, or enterprise SSO.
- Password reset emails.
- User profile management UI.
- Organization or tenant administration UI.
- Fine-grained permissions beyond `admin` and `user`.

## Account Policy

Accounts are created by trusted operators only.

Supported creation paths:

1. `npm run users:create -- --email admin@example.com --name "Admin" --role admin`
2. A seed script for local or staging environments.
3. Direct SQL only when a password hash has already been generated by trusted tooling.

The app must not expose a public `/register` page or `/api/auth/register` endpoint.

## Target Data Model

### `users`

Stores durable user records.

| Column          | Type          | Required | Notes                                   |
| --------------- | ------------- | -------- | --------------------------------------- |
| `id`            | `uuid`        | yes      | Primary key, generated by Postgres.     |
| `email`         | `text`        | yes      | Display email supplied by the operator. |
| `display_name`  | `text`        | yes      | Human-readable name.                    |
| `password_hash` | `text`        | yes      | Bcrypt or Argon2id hash only.           |
| `role`          | `text`        | yes      | `admin` or `user`; default `user`.      |
| `is_active`     | `boolean`     | yes      | Inactive users cannot sign in.          |
| `last_login_at` | `timestamptz` | no       | Updated after successful login.         |
| `created_at`    | `timestamptz` | yes      | Default `now()`.                        |
| `updated_at`    | `timestamptz` | yes      | Updated when account fields change.     |

Indexes and constraints:

- Primary key on `id`.
- Unique case-insensitive email constraint, preferably `unique (lower(email))`.
- Check constraint on `role in ('admin', 'user')`.
- Index on `is_active` for administrative listings and audits.

The existing `002_users_table.sql` migration should be revised or followed by a new migration that
adds the case-insensitive unique email constraint and role check constraint. Before adding a unique
`lower(email)` index to existing data, detect and resolve duplicate emails.

### `auth_sessions`

Stores browser sessions created after password login.

| Column               | Type          | Required | Notes                                       |
| -------------------- | ------------- | -------- | ------------------------------------------- |
| `id`                 | `uuid`        | yes      | Primary key.                                |
| `user_id`            | `uuid`        | yes      | References `users(id)` with cascade delete. |
| `session_token_hash` | `text`        | yes      | SHA-256 hash of random session token.       |
| `expires_at`         | `timestamptz` | yes      | Absolute expiry.                            |
| `created_at`         | `timestamptz` | yes      | Default `now()`.                            |
| `last_seen_at`       | `timestamptz` | yes      | Updated on authenticated activity.          |
| `revoked_at`         | `timestamptz` | no       | Set on logout or admin revocation.          |
| `user_agent`         | `text`        | no       | Request user-agent at login.                |
| `ip_address`         | `inet`        | no       | Client IP when available from the platform. |

Indexes and constraints:

- Unique index on `session_token_hash`.
- Index on `(user_id, expires_at)`.
- Partial index for active session lookup:
  `where revoked_at is null and expires_at > now()` is not valid because `now()` is volatile; use an
  index on `(session_token_hash, revoked_at, expires_at)` instead.
- Cleanup job can delete sessions where `expires_at < now()` or `revoked_at is not null`.

### `login_audit_events`

Records security-relevant account events.

| Column       | Type          | Required | Notes                                                           |
| ------------ | ------------- | -------- | --------------------------------------------------------------- |
| `id`         | `uuid`        | yes      | Primary key.                                                    |
| `user_id`    | `uuid`        | no       | Present when a matching user exists.                            |
| `email`      | `text`        | yes      | Normalized attempted email.                                     |
| `event_type` | `text`        | yes      | `login_success`, `login_failure`, `logout`, `inactive_account`. |
| `success`    | `boolean`     | yes      | Convenience flag for reporting.                                 |
| `reason`     | `text`        | no       | Internal reason; never returned to the client.                  |
| `ip_address` | `inet`        | no       | Client IP when available.                                       |
| `user_agent` | `text`        | no       | Request user-agent.                                             |
| `created_at` | `timestamptz` | yes      | Default `now()`.                                                |

Indexes:

- `(email, created_at desc)` for account-specific review.
- `(ip_address, created_at desc)` for abuse investigation.
- `(created_at desc)` for retention cleanup.

## Password Requirements

- Store only `password_hash`, never plaintext passwords.
- Prefer Argon2id when operationally convenient; bcrypt is acceptable because it is already installed
  in `apps/web`.
- Use a cost factor appropriate for the production host and document the selected value.
- Enforce a minimum password length of 12 characters for operator-created accounts.
- Return generic login failures regardless of whether the email exists.

## Session Cookie Requirements

- Cookie name defaults to `onboarding_session`.
- Cookie is `HttpOnly`, `Secure` in production, `SameSite=Lax`, and `Path=/`.
- Session token is at least 32 random bytes encoded with base64url.
- Only the hash of the token is stored in Postgres.
- Default session duration is 8 hours.
- Logout revokes the database session and clears the cookie.
- Expired or revoked sessions are rejected.

## Runtime Configuration

Add these environment variables:

| Variable                      | Default              | Notes                                                 |
| ----------------------------- | -------------------- | ----------------------------------------------------- |
| `ACCOUNT_AUTH_ENABLED`        | `false`              | Enables Postgres email/password auth.                 |
| `ACCOUNT_SESSION_COOKIE_NAME` | `onboarding_session` | Browser session cookie name.                          |
| `ACCOUNT_SESSION_TTL_SECONDS` | `28800`              | Default 8 hours.                                      |
| `ACCOUNT_COOKIE_SECURE`       | production-only      | Override only for local non-HTTPS development.        |
| `ACCOUNT_PASSWORD_HASH_COST`  | `12`                 | Bcrypt cost if bcrypt remains the selected algorithm. |

When `ACCOUNT_AUTH_ENABLED=true`, require `DATABASE_URL` and the account migrations. The existing
`AUTH_DISABLED`, `API_AUTH_TOKEN`, and JWKS paths may remain available for local development or
trusted gateway deployments, but production should choose one primary authentication mode.

## Authentication Flow

1. User opens `/workspace`.
2. Server checks the `onboarding_session` cookie.
3. If missing or invalid, the user sees the login screen.
4. User submits email and password.
5. `POST /api/auth/login` normalizes the email and looks up the `users` row in Postgres.
6. Server verifies the password hash.
7. Server rejects inactive users and records an audit event.
8. Server creates an `auth_sessions` row with a hashed random token.
9. Server sets the secure session cookie.
10. API route handlers resolve the authenticated user from the cookie.
11. On logout, the current `auth_sessions` row is revoked and the cookie is cleared.

## API Changes

### `POST /api/auth/login`

Request:

```json
{
  "email": "admin@example.com",
  "password": "correct horse battery staple"
}
```

Response:

```json
{
  "user": {
    "id": "uuid",
    "email": "admin@example.com",
    "displayName": "Admin",
    "role": "admin"
  }
}
```

Failure response is always a generic `401`:

```json
{
  "error": "Invalid email or password"
}
```

### `GET /api/auth/me`

Reads the session cookie and returns the current user profile.

### `POST /api/auth/logout`

Revokes the current `auth_sessions` row and clears the cookie.

## UI Changes

- Replace the current Account ID / Access Token login form with Email / Password fields when
  account auth is enabled.
- Keep a deliberate local-development auth mode only when documented in README and production
  validation rejects it.
- Do not store password credentials or session tokens in `sessionStorage`.
- Show generic login errors.
- Display the signed-in user's email or display name in the workspace rail.

## Onboarding Data Ownership

`PostgresSessionRepository` already scopes onboarding sessions by `owner_id`. With account auth,
`owner_id` should be the authenticated `users.id` UUID string.

Required behavior:

- New sessions use `currentUser.id` as `owner_id`.
- Session list, read, update, delete, chat, and guide routes filter by `currentUser.id`.
- File-backed local sessions may continue to use the same interface for local development.
- Existing sessions created by header/JWT auth may need an operator migration if production data
  should be preserved.

## Acceptance Criteria

- Users can sign in with Postgres-backed accounts created by trusted operators.
- Users cannot self-register.
- Passwords and browser session tokens are never stored in plaintext.
- Authenticated browser sessions are stored server-side in Postgres.
- Logout revokes the server-side session.
- Expired and revoked sessions are rejected.
- Protected API routes reject unauthenticated requests.
- Onboarding sessions are scoped by authenticated `users.id`.
- Generic login failures do not reveal whether an email exists.
- Login success, login failure, logout, and inactive-account attempts are audited.
- Production docs clearly explain required migrations and environment variables.
